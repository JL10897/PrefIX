背景：
原来 BFCL 的 turn 都是固定 query，不是自然对话
必须把不同 persona 写死在这些 turn
但是由于我们在model 的是 interaction preference 的不同
turn 写死了就无法模拟真实用户的动态行为

将 BFCL 原多轮 query 聚合成一个 high-level instruction, 即“这个任务的整体目标”, 这个 instruction 不含个性化，也不含 turn 结构。

agent（被测模型）执行工具调用，带着用户 simulator 的下一轮 query。 对于每一个被测模型，simulated user 的行为是 deterministic 的（across different models），任务也一样，但 agent 会因为 history 不同而展现不同行为

必须确保不同模型的 evaluation 时,user simulator 用同一个 checkpoint、同一个 seed、同一个 persona prompt。

## BFCL 现有 turn-by-turn 流程梳理

1. CLI 入口：`bfcl_eval/__main__.py:52-152` 的 `generate()` 用 Typer 接收 `--model`、`--test-category` 等参数，直接调用 `_llm_response_generation.main()`。
2. 载入测试数据：`_llm_response_generation.py:38-121` 中的 `get_involved_test_entries()` / `load_file()` 从 `bfcl_eval/data/*.json` 读出 JSONL，每条 entry 带 `question`（turn-by-turn 提示）、`initial_config`、`involved_classes` 等字段。`process_multi_turn_test_case()` 会根据 `involved_classes` 合并 `data/multi_turn_func_doc` 中的函数文档，并处理 miss-function/miss-param 标记。
3. Handler 调用：`multi_threaded_inference()` 对每个 entry 执行 `handler.inference()`（handler 由 `constants/model_config.py` 注册的类提供）。Function Calling 模型走 `BaseHandler.inference_multi_turn_FC`，Prompting 模型走 `inference_multi_turn_prompting`。
4. Turn 驱动：`model_handler/base_handler.py:118-275` 用 `test_entry[question]` 里的 turn 队列逐轮注入。第一轮调用 `add_first_turn_message_FC()`，后续轮 `_add_next_turn_user_message_FC()`。每轮内部用 `_query_FC()` 发送上下文→`_parse_query_response_FC()` 解析→`decode_execute()`+`execute_multi_turn_func_call()` 执行工具→`_add_execution_results_FC()` 把工具输出接回聊天历史，直到模型停止或 hit `MAXIMUM_STEP_LIMIT`。
5. Prompt 拼装：通用系统提示在 `constants/default_prompts.py`，`DEFAULT_SYSTEM_PROMPT` 会插入函数 JSON。具体 handler 通过 `_compile_tools()`（例如 `api_inference/openai_response.py:100-134`）将函数 schema 转化成目标 API 的 tool 列表；对于 Prompting handler，`system_prompt_pre_processing_chat_model()`（`model_handler/utils.py:326-454`）会把函数文档写入 system prompt。
6. 评测收尾：所有 turn 的模型输出保存在 `<RESULT_PATH>/<model>/<test>_result.json`，`eval_checker/eval_runner.py:19-118` 里 `multi_turn_runner()` 重新读取结果，与 `possible_answer` 中的参考路径比对，调用 `multi_turn_checker()` / `execute_multi_turn_func_call()` 来复现执行并算分。

这条链路就是“/data 中的 turn-by-turn user query 如何进入模型”的完整流程，后续如果要插入 persona LLM，可在 `process_multi_turn_test_case()` 或 handler 的 `_pre_query_processing_FC()` 阶段改写 `test_entry[question]`，其它调用链可保持不变。

Current BFCL user-message flow

- Data source: multi-turn prompts live in gorilla/berkeley-function-call-leaderboard/data/BFCL_v3_multi_turn_*.json (and similar). Each test entry already contains question:
  List[List[dict]] with role-tagged messages per turn, plus function, initial_config, involved_classes, missed_function (optional).
    
- Runner path: CLI bfcl → bfcl_eval/eval_checker/eval_runner.py loads dataset into prompt entries, builds a handler via MODEL_CONFIG_MAPPING, and calls handler.inference(test_entry, include_input_log, exclude_state_log).

- Dispatch: BaseHandler.inference (FC vs prompting) routes to inference_multi_turn_FC or inference_multi_turn_prompting if "multi_turn" in test_entry["id"].

- Preprocessing: inference_multi_turn_FC seeds inference_data via _pre_query_processing_FC (model-specific transforms: role substitution, system prompt extraction, user-merge, etc.) then _compile_tools (convert function docs to model tool schema).

- Turn loop: all_multi_turn_messages = test_entry["question"]; for each turn:
- Miss-function turn may inject placeholder user prompt and recompile tools.
- First turn goes through add_first_turn_message_FC; later turns through _add_next_turn_user_message_FC to append user message(s) into chat history.
- Query: _query_FC(inference_data) hits the model API; _parse_query_response_FC extracts model_responses, token counts, reasoning content, tool_call_ids, etc.; _add_assistant_message_FC appends assistant reply to history.
- Decode/execute: decode_execute(model_responses) → execute_multi_turn_func_call (maps function names to instances, loads state from initial_config, executes, returns strings + involved_instances); _add_execution_results_FC appends tool outputs for the next turn.
- Control/metrics: step-level loop until empty response, decode error, or count > MAXIMUM_STEP_LIMIT (force quit). Collect input_token_count, output_token_count, latency, reasoning_content, inference_log, state_info snapshots (if enabled). all_model_response stores only raw model responses per step/turn; metadata returned separately.
- Prompting path mirrors FC but simulates tool outputs as chat text and skips real tool calls.
